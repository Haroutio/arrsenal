# Arrsenal ‚Äî Design

This document records the design decisions made before the first line of code, and the
reasoning behind them. It is the contract for contributors: PRs that quietly cross one of
these lines will be asked to argue the line first.

## 1. What Arrsenal is (and is not)

**Arrsenal is a declarative regenerator, not a stack manager.**

- The user's answers are the state. One state file (under `/opt/arrsenal/`) holds the
  selected apps, PUID/PGID/TZ/UMASK, paths, GPU choice, and port remaps. The TUI opens
  pre-populated from it on every run.
- `docker-compose.yml` and `.env` are **build artifacts**, regenerated deterministically
  from the state file on every run, with a `GENERATED BY ARRSENAL ‚Äî DO NOT EDIT` header.
  There is never a hand-edited-file diff to reconcile, because hand-edits have a
  first-class home instead:
- **`docker-compose.override.yml` is the escape hatch.** Compose merges it natively;
  Arrsenal never reads or writes it. Power-user customization survives every regeneration.
- **Reconciliation is delegated to Compose.** `docker compose up -d --remove-orphans`
  already is a declarative reconciler: deselected app ‚Üí container removed; changed
  config ‚Üí container recreated. Arrsenal contains zero diffing logic by design.
- **Iron rule: Arrsenal never deletes data.** Deselecting an app removes its container;
  `/opt/appdata/<app>` stays on disk, with a printed note. Removal is always reversible.

Rejected alternatives, for the record:

- *One-shot installer* (generate once, user owns files after): less useful; re-running
  to add/remove apps is the feature people actually want.
- *True manager* (parse and merge user-edited compose files): the DockSTARTer tar pit.
  Reconciling hand-edited YAML is a product-sized problem with no ceiling.

## 2. Architecture

Two components, each doing only what it's good at:

- **`install.sh`** ‚Äî thin bash bootstrap (~150 lines, dumb on purpose). Detects
  distro/arch, offers to install Docker + the compose plugin if missing, downloads the
  right `arrsenal` release binary, verifies its SHA-256 checksum against the release's
  checksums file, and execs it. This is the entire `curl | bash` surface. The README
  one-liner fetches it from `main` (amended v0.4: a version in the URL goes stale
  every release and is hostile to typing by hand). Integrity rests where it always
  did ‚Äî on the checksum-verified, release-built binary ‚Äî and the read-before-you-run
  path runs the exact file you read. The cost is that the script itself is mutable at
  fetch time, which the thin-bootstrap rule (below) is what keeps acceptable:
  install.sh stays dumb, stable, and reviewable in one screenful.
- **`arrsenal`** ‚Äî a single static Go binary. Bubble Tea TUI, state file handling,
  compose/`.env` generation (structs marshalled to YAML ‚Äî no string templating),
  preflight checks, the wiring engine (net/http clients), and the test suite. No runtime
  dependencies.

Go was chosen over the original bash+gum plan deliberately: the product is a CLI with
persistent state, deterministic YAML generation, and a REST-wiring engine ‚Äî the three
things bash is worst at. The "optional Go rewrite later" milestone was recognized as a
fiction (optional rewrites never happen) and deleted; we start where we'd end up.

## 3. The app registry

Every supported app is an entry in a registry that declares:

- `role` ‚Äî `media-server`, `requests`, `indexer`, `pvr`, `subtitles`,
  `download-client`, `dashboard`
- image (LinuxServer.io for the arr suite + download clients; official images where
  they're better maintained), default port, volumes, env
- wiring capability tier: `full-auto` / `semi-auto` / `manual`
- TUI warnings (e.g. paywall notes for Plex/Emby hardware transcode)

The registry is what makes v0.3's Plex/Emby support "two entries and a pairing rule"
instead of a refactor.

**v0.1 menu (10):** Jellyfin, Jellyseerr, Prowlarr, Sonarr, Radarr, Lidarr, Bazarr,
SABnzbd, qBittorrent, Homepage.

Deliberate cuts:

- **Readarr** ‚Äî retired upstream (2025); its metadata backend was already broken.
  Excluded until the ecosystem produces a successor.
- **Pi-hole, Mealie** ‚Äî not media automation. They belong in a user's override file,
  not in the product.
- **Whisparr, second Radarr instances, Tautulli-class extras** ‚Äî later, if the registry
  makes them cheap; never at the cost of the core promise.

**Requests pairing rule:** Jellyfin/Emby ‚Üí Jellyseerr; Plex ‚Üí Overseerr (Jellyseerr
offered as alternative ‚Äî it supports all three servers).

## 4. Brownfield policy: coexistence, not adoption

- **Coexistence (supported in v0.1):** preflight scans every *selected* app for port
  conflicts, existing container names, and non-empty `/opt/appdata/<app>` dirs. Conflicts
  are not fatal: the TUI offers per-app port remaps (persisted in state) or deselection.
  Existing appdata produces a notice: "existing config found ‚Äî the new container will
  adopt it."
- **Adoption (refused):** Arrsenal never imports, migrates, wires, or manages containers
  and installs it did not create. Wiring an unknown instance means unknown version,
  unknown auth, unknown path layout, and a broken hardlink guarantee. The workaround is
  manual and documented: stop the old container, copy its config into
  `/opt/appdata/<app>`, select the app ‚Äî the new container adopts the old database.
- Because adopted configs arrive with their own settings, **every wiring call is
  idempotent by name**: list first, add only what's absent, never modify or duplicate
  an existing entry. This rule is mandatory for safe re-runs anyway; brownfield gets it
  for free.

## 5. Preflight

Runs after the TUI, before anything is written or started:

1. **Conflict scan** ‚Äî ports (host-bound), container names, existing appdata (per ¬ß4).
2. **Paths are prompted, never assumed.** The paths screen shows a live mount table
   ("`/mnt/das` ‚Äî 54 TB free ¬∑ `/` ‚Äî 38 GB free (OS disk)"). Choosing a path on the root
   filesystem triggers an explicit are-you-sure with free-space numbers. Missing
   directories are created only after confirmation, owned `PUID:PGID`.
3. **Hardlink test ‚Äî real, not inferred.** Create a temp file under the downloads dir,
   attempt an actual hardlink into the media dir, clean up. Device-ID heuristics lie
   (mergerfs being the canonical case); the syscall doesn't. Failure = loud warning
   ("imports will copy instead of hardlink") but never a hard block ‚Äî copy-mode is
   legitimate on NFS-backed setups.
4. **Directory tree** ‚Äî the TRaSH layout (`media/{tv,movies,music}`,
   `usenet/{complete,incomplete}`, `torrents/{tv,movies,music}` matching the download
   clients' category dirs) is created with
   correct ownership, and validated for writability, before first boot.
   An optional **downloads root** may split the download trees onto their own
   filesystem (SSD scratch + big-array media): container paths never change, only the
   host side of the mounts, so wiring is unaffected. Splitting trades hardlink imports
   for copy-mode ‚Äî presented as the informed choice it is, not a warning. The
   single-root hardlink layout remains the default and the recommendation. The
   container-internal layout is **fixed** ‚Äî that's the opinion in "opinionated
   installer," and it keeps the wiring engine's root-folder calls deterministic.

## 6. Networking

- **One named bridge network; every wired connection is a container-name URL**
  (`http://sonarr:8989`). No host-IP detection, no interface guessing.
- Jellyfin defaults to bridge with 8096 published. The TUI offers a host-networking
  toggle for DLNA/auto-discovery users; choosing it makes Jellyseerr‚ÜíJellyfin need a
  host address, so the TUI shows the detected LAN IP and asks for confirmation ‚Äî the
  only place a host IP ever appears, and it's user-confirmed.
- Every published port is remappable (preflight feeds the defaults).

## 7. The wiring engine (v0.2 ‚Äî the killer feature)

**Apps own their API keys.** Arrsenal reads keys the apps generate, rather than
pre-seeding its own ‚Äî the read path must exist anyway for brownfield adoption (¬ß4), so
pre-seeding would only add a write path for a determinism win that readiness polling
already provides.

Order of operations:

1. `docker compose up -d` the core apps.
2. Poll each app's config file until its self-generated key appears (per-app timeout,
   failure names the container and the exact `docker logs` command), then poll its API
   to readiness with that key.
3. Set authentication via each arr's host-config API ‚Äî one admin username/password
   collected once in the TUI and applied everywhere, so nobody clicks through five
   identical first-run auth screens. **The admin password is used, not kept** (¬ß9).
4. Wire, idempotently by name: Prowlarr gets each arr registered as an application;
   each arr gets its download client(s) and root folders; Jellyfin gets its `/Startup`
   wizard completed (admin user), libraries created, and hardware transcoding configured
   to match detected hardware.
5. **File-driven apps go last:** Bazarr and Homepage configs are generated once the
   other apps' keys are known, then those containers start. They move to the tail of the
   boot order instead of requiring an API dance.
6. **Wiring report:** a closing table ‚Äî per connection: `‚úì wired` / `‚óè already existed`
   / `‚úó failed (reason + the manual fallback URL)`. Partial failure is not rollback:
   containers stay up; the report tells the user the one thing left to click.

**The single pre-seed exception: qBittorrent.** It generates a random WebUI password
and prints it only to container logs ‚Äî the exact scraping mess this design avoids ‚Äî so
Arrsenal pre-seeds its WebUI password. One exception, documented, not a pattern.

## 8. GPU / hardware transcoding

Three lanes, all ending with Jellyfin's encoder actually configured via API (detection
alone leaves users transcoding on CPU and blaming the tool):

- **NVIDIA:** if `nvidia-smi` works, offer to install nvidia-container-toolkit (repo +
  package + runtime config ‚Äî safe), add the device reservation, set NVENC.
- **Intel:** `/dev/dri` present ‚Üí mount, render group, set QSV.
- **AMD:** `/dev/dri` present ‚Üí mount, render group, set VAAPI.
- **None / declined:** CPU transcode, stated plainly.

**Kernel drivers are never installed. Permanently.** Secure-boot signing, DKMS, and
headless quirks are a support swamp with distro-specific failure modes. If the driver is
missing, Arrsenal says exactly what to install, links the distro's docs, and proceeds ‚Äî
GPU absence never blocks an install.

## 9. Security posture

1. **Keys on disk ‚Äî don't pretend to fix what can't be fixed.** The arr apps store API
   keys in plaintext config; that is the ecosystem. Arrsenal's job is to not make it
   worse: state file and anything holding secrets is `0600` root-owned; secrets never
   appear in stdout, logs, or the wiring report; no telemetry, ever. No vault theater ‚Äî
   a vault whose key sits on the same disk is a lock taped to its own key.
2. **The admin password is used, not kept.** Collected in the TUI, pushed to each app's
   API, then dropped. (qBittorrent's pre-seeded password is the one persisted secret,
   flagged in docs.)
3. **`curl | bash`, hardened and honest.** Checksum-verified binary download from a
   GitHub release, a "download and read it first" path in the README above the fold,
   and the script never asks to be piped into sudo ‚Äî it requests privilege itself, per
   action, announcing why first (see ¬ß10). The script URL points at `main` (amended
   v0.4, ¬ß2): the binary checksum is the integrity boundary, not the script ref.

## 10. Prerequisites & distro tiers

**Inform-then-prompt is universal.** Nothing installs silently; nothing just says
"go away and come back." `install.sh` shows the prerequisite checklist with per-item
prompts (Docker, compose plugin); the binary's GPU lane handles the NVIDIA container
toolkit the same way (¬ß8).

- **Tier 1 ‚Äî Debian 12+, Ubuntu 22.04+ LTS (amd64 + arm64):** full prerequisite
  handling (Docker, compose plugin, NVIDIA toolkit), CI-tested; bugs here block release.
- **Tier 2 ‚Äî any Linux where `docker compose version` already works:** prereq install
  skipped; everything else (generation, preflight, up, wiring) is distro-agnostic and
  fully supported. SELinux-enforcing hosts get a volume-labeling warning rather than a
  mystery failure.
- **Coverable non-Tier-1** (Fedora, RHEL, etc.): Arrsenal offers Docker's own official
  installer (get.docker.com), clearly framed best-effort, default answer **No**. We
  delegate to the vendor's script rather than owning per-distro package logic.
- **Refused with a clear message:** WSL2 and Docker-in-LXC. Prompting can't fix those
  environments; it only moves the failure somewhere harder to debug. The message links
  to a "use a VM" explainer.

Every interactive prompt has a flag equivalent (`--yes`, `--apps ...`); full headless
mode matures in v0.3.

## 11. Naming & distribution

- Repo, binary, and brand share one name: **`arrsenal`** (`github.com/Haroutio/arrsenal`).
  The original plan's split (repo `servarr-stack`, brand "Arrsenal") was dropped:
  `servarr-stack` is a crowded generic namespace (10+ existing repos), and the memorable
  name belongs in the URL people share. Discoverability is what GitHub topics are for.
- Releases: **goreleaser** ‚Äî linux/amd64 + linux/arm64, tagged versions, a checksums
  file that `install.sh` verifies against.
- v0.1 ships **quietly** (tagged, no announcements). v0.2 ‚Äî the wiring ‚Äî is the debut.

## 12. Testing

- **Golden-file tests** for generation: a given state file must produce byte-identical
  `docker-compose.yml` + `.env`. Regressions appear as readable diffs in PRs.
- **Unit tests with fakes** for preflight logic, registry, and wiring-client behavior
  (idempotency: "entry exists ‚Üí no call made" is a test, not a hope).
- **Live integration in CI:** GitHub Actions runners have real Docker. Every PR brings
  up the actual generated stack (minus GPU) and runs the real wiring engine against
  live containers, asserting through the apps' own APIs: Prowlarr lists its
  applications, each arr has its download client and root folders, Jellyfin's wizard is
  complete. The killer feature is proven continuously, not hoped about.
- Full `install.sh` runs on clean Debian/Ubuntu VMs remain a manual pre-release ritual.
